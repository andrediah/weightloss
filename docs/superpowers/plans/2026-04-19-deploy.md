# Deployment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Automate deployment of WeightLossTracker to a Digital Ocean Droplet via GitHub Actions on every push to `main`.

**Architecture:** Push to `main` triggers GitHub Actions, which runs tests, publishes a self-contained linux-x64 binary, copies it to the Droplet via SCP, and restarts a systemd service. nginx sits in front as a reverse proxy handling TLS.

**Tech Stack:** GitHub Actions, appleboy/ssh-action, appleboy/scp-action, systemd, nginx, Let's Encrypt (Certbot), ASP.NET Core 10, SQLite

These steps require human interaction and cannot be automated. Complete them in order before beginning Task 1.

When you see `YOUR_DROPLET_IP` or similar placeholders in ALL CAPS, replace the entire thing — including the surrounding quotes if any — with your real value before pressing Enter.

### Pre-1: Generate a deploy SSH keypair (on your local machine)

An SSH keypair is like a lock and key. You keep the private key on your machine; the public key goes on the server. This lets GitHub Actions connect to your server securely without a password.

**Open a terminal on your local machine** (on Windows, open Git Bash or PowerShell), then run:

```bash
ssh-keygen -t ed25519 -C "weightloss-deploy" -f ~/.ssh/weightloss_deploy
```

When it asks `Enter passphrase`, just press **Enter** twice (no passphrase needed).

Now copy the two key files so you have them ready later:

```bash
cat ~/.ssh/weightloss_deploy.pub
```
Select all the output and copy it — you will need it in **Pre-3, Step 4**.

```bash
cat ~/.ssh/weightloss_deploy
```
Select all the output (including the `-----BEGIN OPENSSH PRIVATE KEY-----` and `-----END OPENSSH PRIVATE KEY-----` lines) and copy it — you will need it in **Pre-4**.

### Pre-2: Create a Digital Ocean Droplet (your cloud server)

A Droplet is a small virtual server in the cloud that will host your app.

1. Go to [digitalocean.com](https://digitalocean.com) and log in.
2. Click **Create → Droplets** in the top menu.
3. Under **Choose an image**, select **Ubuntu 24.04 LTS**.
4. Under **Choose a plan**, select **Basic** → **$6/mo** (1 vCPU, 1 GB RAM is enough).
5. Under **Choose a datacenter region**, pick the one closest to you.
6. Under **Authentication**, choose **SSH Key** and add your **personal** SSH public key (the one you normally use to log into servers — not the `weightloss_deploy` key you just created).
7. Click **Create Droplet**.
8. Wait about 30 seconds, then note the **IP address** shown — you will use it throughout the rest of these steps.

**Step 5 — Allow the app to restart itself**

This lets the app user restart the service without needing a password.

> **Important:** sudo parses every file in `/etc/sudoers.d/` on every invocation. A syntax error in this file disables all NOPASSWD rules and deploys will fail with `sudo: a password is required`. The block below uses sudoers' backslash line-continuation (`\`) so each physical line is short — terminals cannot wrap-break a short line during paste, which is the most common way this file ends up broken.

Paste this entire block as one unit:

```bash
rm -f /etc/sudoers.d/weightloss
cat > /etc/sudoers.d/weightloss << 'EOF'
weightloss ALL=(ALL) NOPASSWD: \
    /usr/bin/systemctl restart weightloss.service, \
    /usr/bin/systemctl is-active weightloss.service
EOF
chmod 440 /etc/sudoers.d/weightloss
visudo -c -f /etc/sudoers.d/weightloss
```

The last command **must print** `/etc/sudoers.d/weightloss: parsed OK`. If it prints a syntax error, re-run the block — do not leave a broken file in place.

**To reset** after a broken attempt, run the same block again: the `rm -f` at the top guarantees a clean rewrite. Then confirm with:

```bash
cat -A /etc/sudoers.d/weightloss
```

`cat -A` shows every newline as `$`; the file should have exactly four `$` markers (three line ends + trailing newline), not more.

---

**Step 6 — Create the systemd service**

This tells the server how to run your app and restart it automatically if it crashes.

**Before running this, replace the two placeholders:**
- `REPLACE_WITH_GEMINI_API_KEY` → your actual Gemini API key
- `REPLACE_WITH_ADMIN_API_KEY` → your actual Admin API key

```bash
cat > /etc/systemd/system/weightloss.service << 'EOF'
[Unit]
Description=WeightLossTracker
After=network.target

[Service]
User=weightloss
WorkingDirectory=/opt/weightloss
ExecStart=/opt/weightloss/WeightLossTracker
Restart=always
RestartSec=5

Environment=ASPNETCORE_ENVIRONMENT=Production
Environment=ASPNETCORE_URLS=http://localhost:5000
Environment=Gemini__ApiKey=REPLACE_WITH_GEMINI_API_KEY
Environment=Admin__ApiKey=REPLACE_WITH_ADMIN_API_KEY
Environment="ConnectionStrings__DefaultConnection=Data Source=/var/lib/weightloss/weightloss.db"

[Install]
WantedBy=multi-user.target
EOF
```

> **Important:** The `<< 'EOF'` syntax means everything between the two `EOF` markers is written to the file. You must replace the placeholder values inside the block before running it.

> **Why the quotes around `ConnectionStrings__DefaultConnection`:** systemd `Environment=` splits on whitespace unless the whole assignment is double-quoted. The SQLite connection string contains a space (`Data Source=…`), so without quotes systemd would only pass the literal value `Data` to the app and the SQLite driver would throw `Format of the initialization string does not conform to specification`.

Then lock down the file and enable the service:

```bash
chmod 600 /etc/systemd/system/weightloss.service
chown root:root /etc/systemd/system/weightloss.service
systemctl daemon-reload
systemctl enable weightloss.service
```

**Step 7 — Configure nginx**

nginx acts as a "front door" — it receives traffic on port 80/443 and forwards it to your app on port 5000.

> **Why `default_server` and `server_name _;`:** Ubuntu ships with a default nginx site that replies "Welcome to nginx" to any host not explicitly matched. If you visit the Droplet by IP (or before DNS has propagated), you'll hit that default page instead of your app. Marking the weightloss config as `default_server` and using `_` as a catch-all `server_name` makes it handle everything that isn't explicitly routed elsewhere.

Write the site config (no replacements needed):

```bash
cat > /etc/nginx/sites-available/weightloss << 'EOF'
server {
    listen 80 default_server;
    server_name _;

    location / {
        proxy_pass http://localhost:5000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection keep-alive;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
EOF
```

Activate the config and reload nginx (each line short so terminals cannot wrap-break them):

```bash
rm -f /etc/nginx/sites-enabled/default
rm -f /etc/nginx/sites-enabled/weightloss
ln -s /etc/nginx/sites-available/weightloss /etc/nginx/sites-enabled/weightloss
nginx -t
systemctl reload nginx
```

`nginx -t` should print `syntax is ok` and `test is successful`. Verify end-to-end:

```bash
curl -sI http://localhost:5000 | head -1
curl -sI http://localhost/      | head -1
```

Both should return `HTTP/1.1 200 OK` or `HTTP/1.1 302 Found` (302 means the app is redirecting unauthenticated users to the login page — that's normal).

## Task 2: Create GitHub Actions deploy workflow

**Files:**
- Create: `.github/workflows/deploy.yml`

- [ ] **Step 1: Create the workflows directory**

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Create `.github/workflows/deploy.yml` with this exact content**

```yaml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup .NET 10
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '10.x'

      - name: Run tests
        run: dotnet test WeightLossTracker.Tests/WeightLossTracker.Tests.csproj --configuration Release --no-restore

      - name: Publish self-contained linux-x64 binary
        run: dotnet publish WeightLossTracker/WeightLossTracker.csproj -c Release -r linux-x64 --self-contained true -o ./publish

      - name: Ensure deploy directory exists on Droplet
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DO_SSH_HOST }}
          username: ${{ secrets.DO_SSH_USER }}
          key: ${{ secrets.DO_SSH_KEY }}
          script: mkdir -p /opt/weightloss

      - name: Copy published files to Droplet
        uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.DO_SSH_HOST }}
          username: ${{ secrets.DO_SSH_USER }}
          key: ${{ secrets.DO_SSH_KEY }}
          source: "./publish/*"
          target: "/opt/weightloss"
          strip_components: 1

      - name: Restart service and verify
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.DO_SSH_HOST }}
          username: ${{ secrets.DO_SSH_USER }}
          key: ${{ secrets.DO_SSH_KEY }}
          script: |
            set -e
            sudo /usr/bin/systemctl restart weightloss.service
            sleep 8
            state=$(sudo /usr/bin/systemctl is-active weightloss.service || true)
            echo "service state: $state"
            [ "$state" = "active" ]
```

> **Why the exact command paths matter:** sudoers matches the command path **literally**, not by symlink resolution. The sudoers rule from Pre-3 Step 5 lists `/usr/bin/systemctl`, so the workflow must also call `/usr/bin/systemctl`. It also lists no extra flags, so passing `--quiet` or `status --no-pager` would fall outside the rule and trigger `sudo: a password is required`.

## Troubleshooting & Recovery

If any step goes wrong — or if a partially-applied setup needs to be repaired — match the symptom below to the fix. Each fix is idempotent: running it on a broken *or* healthy system leaves you in the same good state.

### Symptom: Deploy fails with `sudo: a password is required`

The sudoers file on the Droplet has a parse error (usually because a long paste got wrapped into two lines by the terminal). **Reset:** re-run Pre-3 Step 5 exactly as written — it begins with `rm -f /etc/sudoers.d/weightloss` so it cleanly rewrites any broken file. The final `visudo -c` must print `parsed OK`.

Diagnose further with:
```bash
cat -A /etc/sudoers.d/weightloss
```
You should see backslash-newline (`\$`) at the end of the first two lines only; no other `$` in the middle of a line.

### Symptom: Service restarts forever with `Format of the initialization string does not conform to specification`

The `ConnectionStrings__DefaultConnection` line in the systemd unit lost its quotes, so systemd split the value on whitespace and passed only `Data` to the app. **Reset:**

```bash
sudo sed -i 's|^Environment=ConnectionStrings__DefaultConnection=\(.*\)$|Environment="ConnectionStrings__DefaultConnection=\1"|' /etc/systemd/system/weightloss.service
sudo grep ConnectionStrings /etc/systemd/system/weightloss.service
sudo systemctl daemon-reload
sudo systemctl restart weightloss.service
```
The `grep` line should now show the value wrapped in double quotes.

### Symptom: Service restarts forever with `The file '/opt/weightloss/WeightLossTracker' already exists`

This was an old-code bug where the app ignored `ConnectionStrings__DefaultConnection` and computed a DB path from `LocalApplicationData` — which resolved to the working directory when the system user had no `$HOME`. It is fixed in `WeightLossTracker/Program.cs`; deploying any build from `main` after the fix lands clears it. No server change needed.

### Symptom: Browser shows "Welcome to nginx" instead of the app

The Ubuntu default site is still enabled and matches first. **Reset:**

```bash
sudo rm -f /etc/nginx/sites-enabled/default
sudo rm -f /etc/nginx/sites-enabled/weightloss
sudo ln -s /etc/nginx/sites-available/weightloss /etc/nginx/sites-enabled/weightloss
sudo nginx -t
sudo systemctl reload nginx
```
Then re-check that the weightloss site has `listen 80 default_server;` and `server_name _;` — see Pre-3 Step 7.

### Symptom: Users get signed out every time the service restarts

ASP.NET Core's Data Protection keys are not being persisted. The fix lives in `WeightLossTracker/Program.cs` (configures `PersistKeysToFileSystem`). On the Droplet, the app writes keys to a `dp-keys/` directory next to the SQLite DB (i.e., `/var/lib/weightloss/dp-keys/`). Ensure that parent directory is writable by the `weightloss` user — Pre-3 Step 3 sets this up correctly.

To verify after a deploy:
```bash
sudo ls -la /var/lib/weightloss/dp-keys
```
You should see one or more `key-*.xml` files owned by `weightloss:weightloss`.

### Full teardown and rebuild (nuclear option)

If the Droplet state is beyond diagnosing, reset the server-side config without losing the database:

```bash
# 1. Stop and disable the service
sudo systemctl stop weightloss.service
sudo systemctl disable weightloss.service

# 2. Remove managed config (database stays in /var/lib/weightloss)
sudo rm -f /etc/sudoers.d/weightloss
sudo rm -f /etc/systemd/system/weightloss.service
sudo rm -f /etc/nginx/sites-enabled/weightloss
sudo rm -f /etc/nginx/sites-available/weightloss
sudo systemctl daemon-reload
sudo systemctl reload nginx

# 3. Re-run Pre-3 Steps 5, 6, 7 from this document
```

The `weightloss` user, `/opt/weightloss`, `/var/lib/weightloss`, and the deploy SSH key are preserved — only the broken unit/sudoers/nginx config is removed.
